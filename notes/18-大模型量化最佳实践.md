## 1. 量化概述

### 什么是量化？

量化是将模型权重和/或激活值从高精度浮点数（如 FP32、FP16、BF16）转换为低精度表示（如 INT8、INT4、FP8、FP4）的过程。其核心目标是：

- **减少显存占用**：INT4 量化可将模型大小缩减至 FP16 的 1/4
- **提升推理速度**：低精度运算在现代 GPU 上吞吐量更高
- **降低部署成本**：相同硬件可部署更大的模型

### 量化的基本原理

```
量化公式：Q(x) = round(x / scale) + zero_point
反量化公式：x̂ = (Q(x) - zero_point) × scale
```

- **scale（缩放因子）**：将浮点范围映射到整数范围
- **zero_point（零点）**：对称量化时为 0，非对称量化时非零
- **对称量化**：仅使用 scale，零点固定为 0，实现简单
- **非对称量化**：同时使用 scale 和 zero_point，精度更高但计算复杂

### 量化粒度

| 粒度 | 说明 | 精度 | 开销 |
|------|------|------|------|
| **Per-tensor** | 整个张量共享一个 scale | 最低 | 最小 |
| **Per-channel** | 每个输出通道一个 scale | 中等 | 较小 |
| **Per-token** | 每个 token 一个 scale | 较高 | 中等 |
| **Per-group** | 每 group_size 个元素一个 scale（如 128） | 最高 | 较大 |
| **Per-block** | 二维分块，每块一个 scale（如 128×128） | 高 | 中等 |

## 2. 量化技术分类：PTQ vs QAT

### PTQ（训练后量化）

在模型训练完成后直接进行量化，不需要重新训练。

**优点**：
- 实施快速，通常几小时内完成
- 不需要原始训练数据（仅需少量校准数据）
- 工具链成熟，开箱即用

**缺点**：
- 在极低比特（2-3 bit）下精度损失较大
- 对激活异常值（outliers）敏感

**适用场景**：
- 8 bit 和 4 bit 量化（精度损失可控）
- 快速迭代和原型验证
- 缺乏训练数据或训练资源的场景

**代表方法**：GPTQ、AWQ、SmoothQuant、RTN（Round-to-Nearest）

### QAT（量化感知训练）

在训练过程中插入伪量化操作，让模型学习适应低精度表示。

**优点**：
- 在低比特下精度显著优于 PTQ
- 模型主动学习补偿量化误差

**缺点**：
- 需要原始训练数据和计算资源
- 训练时间长，实施成本高

**适用场景**：
- 2-3 bit 极低比特量化
- 精度要求极高的生产环境
- 有充足训练资源的团队

**代表方法**：EfficientQAT、SiLQ、SAPR

### 选择建议

```
决策路径：
1. 先尝试 PTQ（快速验证）
2. 如果 PTQ 精度不满足要求 → 尝试 QAT
3. 如果没有训练数据 → 使用 AWQ（效果最好的 PTQ 方法之一）
4. 如果有训练数据且需要极低比特 → 使用 QAT + SAPR
```

## 3. 主流量化格式详解

### 3.1 FP8（8 位浮点）

```
FP8-E4M3：4 位指数 + 3 位尾数，动态范围大，适合权重和激活
FP8-E5M2：5 位指数 + 2 位尾数，更大动态范围，适合梯度
```

- **优势**：保留浮点数的指数结构，天然处理激活异常值
- **精度损失**：< 2%（几乎无感知）
- **显存节省**：相比 FP16 减少 50%
- **硬件要求**：NVIDIA Hopper (H100/H200)、Ada Lovelace (RTX 4090/A6000)
- **vLLM 支持**：`--quantization fp8`，支持 W8A8 和 W8A16

### 3.2 INT8（8 位整数）

```
范围：[-128, 127]（有符号）或 [0, 255]（无符号）
```

- **优势**：硬件支持广泛，计算效率高
- **精度损失**：< 2%
- **显存节省**：50%
- **硬件要求**：NVIDIA Turing+ (SM75+)
- **vLLM 支持**：`--quantization compressed-tensors`（W8A8 方案）

### 3.3 INT4（4 位整数）

```
范围：[-8, 7]（有符号）
通常配合 group_size=128 的分组量化使用
```

- **优势**：显存节省 75%，性能与精度平衡最佳
- **精度损失**：2-8%（取决于模型大小和方法）
- **硬件要求**：广泛支持
- **vLLM 支持**：通过 GPTQ、AWQ 等方法实现

### 3.4 FP4（4 位浮点）/ NVFP4

```
NVIDIA 专有 FP4 格式，每 16 个元素共享一个 FP8 缩放因子
```

- **优势**：保留浮点特性，比 INT4 更好处理异常值
- **精度损失**：略优于 INT4
- **硬件要求**：NVIDIA Blackwell (SM100) 原生支持；Hopper 需软件模拟
- **vLLM 支持**：`--quantization modelopt_fp4` 或 `compressed-tensors`

### 3.5 MXFP8（微缩放 FP8）

```
OCP（开放计算项目）标准，块级缩放
每 32 个元素共享一个 E8M0 缩放因子
```

- **优势**：细粒度缩放，误差控制更精确
- **精度损失**：< 0.1%（接近无损）
- **硬件要求**：NVIDIA Blackwell 原生支持
- **vLLM 支持**：`--quantization modelopt_mxfp8`

### 3.6 GPTQ

```
基于 Hessian 矩阵的逐层最优量化
支持 2/3/4/8 bit
```

- **原理**：利用二阶（Hessian）信息最小化每层量化误差
- **优势**：通用性强，支持多种比特位宽
- **精度**：4 bit 下接近 FP16
- **推理加速**：配合 Marlin 内核可获得 2.6 倍加速
- **vLLM 支持**：`--quantization gptq` 或 `--quantization gptq_marlin`

### 3.7 AWQ（激活感知权重量化）

```
保护对激活值影响最大的 1% 显著权重
仅量化权重，激活保持 FP16
```

- **原理**：通过观察激活分布，找到最重要的权重通道并给予特殊保护
- **优势**：4 bit 量化精度通常优于 GPTQ
- **推理加速**：配合 Marlin 内核可获得 10.9 倍加速
- **vLLM 支持**：`--quantization awq` 或 `--quantization awq_marlin`

### 3.8 GGUF

```
GGML 统一格式，支持多种量化类型
常用：Q4_K_M、Q5_K_M、Q8_0
```

- **优势**：CPU 推理友好，社区生态丰富
- **适用场景**：本地部署、边缘设备
- **vLLM 支持**：`--quantization gguf`

### 格式对比总结

| 格式 | 比特数 | 压缩比 | 精度损失 | 最佳场景 |
|------|--------|--------|----------|----------|
| FP8 | 8 | 2× | < 2% | GPU 生产环境（Hopper/Ada） |
| INT8 | 8 | 2× | < 2% | 通用 GPU 推理 |
| INT4/GPTQ | 4 | 4× | 2-8% | GPU 推理（显存受限） |
| AWQ | 4 | 4× | 2-5% | GPU 推理（精度优先） |
| FP4/NVFP4 | 4 | 4× | 2-6% | Blackwell GPU |
| MXFP8 | 8 | 2× | < 0.1% | Blackwell GPU（接近无损） |
| GGUF | 4-8 | 2-4× | 变化 | CPU/本地推理 |

## 4. vLLM 支持的量化方法全景

vLLM 支持 **36+ 种量化方法**，是开源推理框架中量化支持最全面的。以下按类别列出：

### 4.1 权重量化（Weight-Only）

| 方法 | 配置名 | 比特数 | 加速内核 | GPU 最低要求 |
|------|--------|--------|----------|-------------|
| AWQ | `awq` | 4 | 原生 | SM75 (Turing) |
| AWQ + Marlin | `awq_marlin` | 4 | Marlin | SM75+ |
| GPTQ | `gptq` | 2/3/4/8 | 原生 | SM70+ |
| GPTQ + Marlin | `gptq_marlin` | 4 | Marlin | SM80+ |
| BitsAndBytes | `bitsandbytes` | 4/8 | BNB | 广泛 |
| GGUF | `gguf` | 多种 | GGML | SM60+ |
| AutoRound/INC | `inc` | 2/3/4/8 | Marlin 可选 | SM70+ |

### 4.2 权重 + 激活量化（W8A8 / W4A8）

| 方法 | 配置名 | 方案 | 加速内核 | GPU 最低要求 |
|------|--------|------|----------|-------------|
| FP8 | `fp8` | W8A8 | Cutlass/Marlin | SM80 (Ampere) |
| FBGEMM FP8 | `fbgemm_fp8` | W8A8 | FBGEMM/Marlin | SM80+ |
| Compressed-Tensors | `compressed-tensors` | W8A8/W4A8 | 多种 | SM70+ |
| Quark | `quark` | W8A8/W4A8 | 多种 | SM70+ |
| ModelOpt | `modelopt` | W8A8 | TRT-LLM | SM80+ |

### 4.3 FP4/MXFP 量化

| 方法 | 配置名 | 格式 | GPU 最低要求 |
|------|--------|------|-------------|
| MXFP4 | `mxfp4` | MX-FP4 | SM90 (Hopper) |
| ModelOpt FP4 | `modelopt_fp4` | NVFP4 | SM90+ |
| ModelOpt MXFP8 | `modelopt_mxfp8` | MX-FP8 | SM90+ |
| FP-Quant | `fp_quant` | MXFP4 | SM100 (Blackwell) |

### 4.4 MoE 专用量化

| 方法 | 配置名 | 说明 |
|------|--------|------|
| MoeWNA16 | `moe_wna16` | MoE 专用 4/8 bit 量化 |
| ExpertsInt8 | `experts_int8` | MoE 专家 INT8 量化（已弃用） |

### 4.5 CPU 专用量化

| 方法 | 配置名 | 说明 |
|------|--------|------|
| CPU AWQ | `cpu_wna16` | CPU 端 AWQ 量化 |
| oneDNN INT8 | 通过代码调用 | Intel oneDNN 加速 |

### 使用方式

```bash
# 命令行指定量化方法

# Python API
from vllm import LLM
llm = LLM(model="path/to/model", quantization="awq")
```

## 5. 如何选择合适的量化方法

### 5.1 按模型大小选择

```
┌─────────────────────────────────────────────────┐
│                模型大小决策树                      │
├─────────────────────────────────────────────────┤
│                                                 │
│  7B 模型                                        │
│  ├── 推荐：AWQ 4-bit（精度损失最小）                │
│  ├── 备选：GPTQ 4-bit                            │
│  └── 避免：2-bit（精度损失过大）                    │
│                                                 │
│  13B 模型                                       │
│  ├── 推荐：AWQ 4-bit（稳定可靠）                   │
│  ├── 备选：FP8（如有 Hopper GPU）                  │
│  └── 注意：GPTQ 在 13B 上精度退化比 AWQ 明显        │
│                                                 │
│  70B+ 模型                                      │
│  ├── 推荐：FP8 W8A8（精度几乎无损）                 │
│  ├── 备选：AWQ/GPTQ 4-bit（均可稳定工作）            │
│  └── 可考虑：3-bit（大模型对低比特更鲁棒）            │
│                                                 │
│  MoE 模型（如 DeepSeek V3/Mixtral）                │
│  ├── 推荐：Quark W8A8-FP8 或 MXFP4                │
│  ├── 备选：moe_wna16（专用优化）                    │
│  └── 注意：需要支持 MoE 的量化工具                   │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 5.2 按硬件选择

| 硬件 | 首选方法 | 备选方法 | 说明 |
|------|---------|---------|------|
| H100/H200 | FP8 W8A8 | MXFP8, AWQ | 原生 FP8 支持，性能最优 |
| A100/A6000 | INT8 W8A8 | AWQ + Marlin | 无原生 FP8，使用 Marlin 加速 |
| RTX 4090 | AWQ + Marlin | GPTQ + Marlin | Ada 架构，Marlin 内核高效 |
| RTX 3090 | AWQ 4-bit | GPTQ 4-bit | Ampere 消费级 |
| Blackwell | MXFP8 / FP4 | FP8 | 原生 FP4 和 MXFP8 支持 |
| AMD MI300X | PTPC-FP8 | GPTQ (CK 后端) | ROCm 专用优化 |
| CPU (Intel) | GGUF Q4_K_M | oneDNN INT8 | CPU 推理首选 GGUF |
| 边缘设备 | GGUF Q4_K_S | INT4 | 极致压缩 |

### 5.3 按精度要求选择

| 精度要求 | 推荐方法 | 典型损失 |
|---------|---------|---------|
| 几乎无损（< 1%） | FP8 W8A8 / MXFP8 | < 0.1-2% |
| 轻微损失（< 5%） | AWQ 4-bit / INT8 | 2-5% |
| 可接受损失（< 10%） | GPTQ 4-bit / 3-bit | 5-10% |
| 激进压缩 | 2-bit QAT | 10-30% |

### 5.4 快速决策流程

```
Q1: 有 Hopper/Ada GPU 吗？
  ├── 是 → 使用 FP8 W8A8（最佳性价比）
  └── 否 → Q2

Q2: 需要 4 bit 压缩吗？
  ├── 是 → Q3
  └── 否 → 使用 INT8 W8A8

Q3: 有校准数据吗？
  ├── 是 → AWQ（精度更好）
  └── 否 → GPTQ（RTN 作为快速备选）

Q4: 部署在 CPU 上吗？
  ├── 是 → GGUF Q4_K_M
  └── 否 → 根据 GPU 选择 Marlin 加速方案
```

## 7. 硬件适配矩阵

### 7.1 NVIDIA GPU 架构支持

| GPU 架构 | 计算能力 | FP8 W8A8 | FP8 W8A16 | INT8 W8A8 | INT4 | FP4 | MXFP8 |
|----------|---------|----------|-----------|-----------|------|-----|-------|
| Volta (V100) | SM70 | - | - | - | GPTQ | - | - |
| Turing (RTX 20) | SM75 | - | Marlin | - | GPTQ/AWQ | - | - |
| Ampere (A100/RTX 30) | SM80-86 | - | Marlin | Cutlass | GPTQ/AWQ | - | - |
| Ada Lovelace (RTX 40) | SM89 | Cutlass | Marlin | Cutlass | GPTQ/AWQ | - | - |
| Hopper (H100/H200) | SM90 | Cutlass | Marlin | Cutlass | GPTQ/AWQ | 软件 | 软件 |
| Blackwell (B200) | SM100 | MXFP8 替代 | Marlin | W8A16 | GPTQ/AWQ | 原生 | 原生 |

### 7.2 显存需求估算

```
模型显存 ≈ 参数量 × 每参数字节数 + KV Cache + 激活值 + 框架开销

每参数字节数：
  FP32 = 4 bytes    FP16/BF16 = 2 bytes
  FP8/INT8 = 1 byte    INT4 = 0.5 bytes
  FP4 = 0.5 bytes

示例（7B 模型）：
  FP16：7B × 2 = 14 GB
  INT8：7B × 1 = 7 GB
  INT4：7B × 0.5 = 3.5 GB
  INT4 + KV Cache + 开销 ≈ 5-6 GB（可在 8GB GPU 上运行）

示例（70B 模型）：
  FP16：70B × 2 = 140 GB（需要多卡）
  INT8：70B × 1 = 70 GB（单卡 A100 80GB 勉强可用）
  INT4：70B × 0.5 = 35 GB（单卡 A100 40GB 可用）
```

### 7.3 AMD GPU 支持

| GPU | 支持的量化方法 |
|-----|--------------|
| MI300X (gfx942) | PTPC-FP8、GPTQ (CK 后端)、MXFP4 (CK) |
| MI250 | GPTQ、AWQ |

### 7.4 CPU 支持

| 平台 | 支持的量化方法 | 加速库 |
|------|--------------|-------|
| Intel (AVX-512) | INT8、INT4 (GGUF) | oneDNN |
| ARM (NEON) | INT8、INT4 (GGUF) | ARM ACL |

## 8. 量化完整工作流程

### 阶段一：环境准备

```bash
# 1. 创建虚拟环境
uv venv --python 3.12
source .venv/bin/activate

# 2. 安装 vLLM
VLLM_USE_PRECOMPILED=1 uv pip install -e .

# 3. 安装量化工具
pip install llmcompressor  # 官方推荐
pip install auto-gptq autoawq  # 备选
```

### 阶段二：方法选择

根据第 5 节的决策流程，确定：
1. 目标比特位宽（8/4/2 bit）
2. 量化方法（AWQ/GPTQ/FP8 等）
3. 量化粒度（per-tensor/per-channel/per-group）

### 阶段三：准备校准数据

```python
# 校准数据集要求：
# - 数量：128-512 个样本（通常 128 即可）
# - 质量：与实际推理任务分布一致
# - 多样性：覆盖不同主题和长度

from datasets import load_dataset

# 示例：使用 C4 数据集作为校准集
calibration_data = load_dataset(
    "allenai/c4",
    data_files="en/c4-train.00000-of-01024.json.gz",
    split="train[:128]"
)
```

### 阶段四：执行量化

**方式 A：使用 LLM Compressor（推荐）**

```python
from llmcompressor.modifiers.quantization import GPTQModifier
from llmcompressor.transformers import oneshot

# FP8 W8A8 动态量化（无需校准数据）
oneshot(
    model="meta-llama/Llama-3.1-8B-Instruct",
    recipe="FP8_DYNAMIC",
    output_dir="./llama-8b-fp8",
)

# GPTQ W4A16 量化（需要校准数据）
recipe = GPTQModifier(
    targets="Linear",
    scheme="W4A16",
    ignore=["lm_head"],
)
oneshot(
    model="meta-llama/Llama-3.1-8B-Instruct",
    recipe=recipe,
    dataset="ultrachat_200k",
    output_dir="./llama-8b-w4a16",
    num_calibration_samples=128,
    max_seq_length=2048,
)
```

**方式 B：使用 AutoAWQ**

```python
from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer

model_path = "meta-llama/Llama-3.1-8B-Instruct"
quant_path = "./llama-8b-awq"

# 加载模型
model = AutoAWQForCausalLM.from_pretrained(model_path)
```

## 9. 精度与性能权衡

### 9.1 不同比特位宽的精度损失

| 比特数 | 典型精度损失 | 对各任务的影响 | 推荐场景 |
|--------|------------|--------------|---------|
| **8-bit** | < 2% perplexity 增加 | 所有任务几乎无影响 | 生产环境标准 |
| **4-bit** | 2-8% 退化 | 代码：98.9% 恢复率；数学：稳定 | 显存受限的生产环境 |
| **3-bit** | 8-15% 退化 | 推理任务受影响明显 | 研究/成本敏感场景 |
| **2-bit** | 15-30% 退化 | 代码/STEM 严重退化 | 仅配合 QAT 使用 |

### 9.2 任务敏感度分析

```
任务敏感度排名（从高到低）：

1. 低资源语言生成    ← 量化影响是高资源语言的 2-4 倍
2. 复杂推理 (BBH)   ← 对量化较敏感
3. 长上下文理解      ← 量化 + 长上下文双重挑战
4. 代码生成          ← 4-bit 下 98.9% 恢复率
5. 数学推理 (GSM8K)  ← 惊人地稳定，即使 4-bit
6. 知识问答 (MMLU)   ← 相对稳定
```

### 9.3 推理性能对比

基于社区基准测试数据（仅供参考）：

| 方法 | 输出吞吐量 | 质量评分 | 说明 |
|------|-----------|---------|------|
| Marlin-AWQ 4-bit | ~741 tok/s | 6.84/10 | 最快，质量好 |
| Marlin-GPTQ 4-bit | ~712 tok/s | 6.80/10 | 次快 |
| FP8 W8A8 | ~600 tok/s | 7.20/10 | 质量最高 |
| GGUF Q4_K_M (vLLM) | ~93 tok/s | 6.74/10 | vLLM 下较慢 |
| FP16 基线 | ~400 tok/s | 7.25/10 | 基线参考 |

> 注：具体性能取决于硬件、模型大小、batch size 等因素。

### 9.4 关键发现

1. **模型越大，量化越鲁棒**：70B 模型在 4-bit 下精度损失远小于 7B
2. **AWQ 通常优于 GPTQ**：在 4-bit 下 AWQ 的精度保持更好
3. **Marlin 内核是性能关键**：AWQ + Marlin 比原生 AWQ 快 10.9×
4. **FP8 是 8-bit 最佳选择**：比 INT8 更好处理激活异常值
5. **训练不充分的模型不适合量化**：训练 token 数 < 100B 的模型量化效果差

## 10. KV Cache 量化

### 10.1 为什么需要 KV Cache 量化？

```
KV Cache 显存 = batch_size × seq_len × num_layers × 2 × hidden_dim × dtype_size

示例（Llama-70B，4096 序列长度，batch=32）：
  FP16 KV Cache ≈ 32 × 4096 × 80 × 2 × 8192 × 2 bytes ≈ 160 GB
  FP8  KV Cache ≈ 80 GB（减半）
  INT4 KV Cache ≈ 40 GB（减 75%）
```

长上下文场景下，KV Cache 可能成为显存瓶颈。

### 10.2 vLLM 的 KV Cache 量化支持

```python
# 方式 1：通过量化配置自动启用 KV Cache 量化
# FP8 量化方法默认支持 KV Cache FP8 量化

# 方式 2：单独配置 KV Cache 量化
# 在 quantization_config 中设置 kv_cache_quant_algo
```

vLLM 中 KV Cache 相关操作（见 `_custom_ops.py`）：
- `reshape_and_cache`：将 K/V 写入分页缓存
- `reshape_and_cache_flash`：Flash Attention 风格的缓存写入
- `concat_and_cache_mla`：MLA 架构的缓存拼接
- `convert_fp8`：FP8 格式转换
- `gather_and_maybe_dequant_cache`：读取缓存并可能反量化

### 10.3 KV Cache 量化建议

| 方法 | 精度损失 | 适用场景 |
|------|---------|---------|
| FP8 KV Cache | < 0.1 perplexity | 长上下文 + GPU 推理 |
| INT4 KV Cache (KVQuant) | < 0.1 perplexity (3-bit) | 极长上下文（10M tokens） |
| 混合精度 KV Cache | 最小化损失 | 不同层使用不同精度 |

## 11. 常见陷阱与规避策略

### 陷阱 1：激活异常值导致量化失败

**现象**：GLU（门控线性单元）结构的模型量化后精度暴跌 15-30%

**原因**：部分激活通道存在极大异常值，量化时破坏整个张量的精度

**解决方案**：
- 使用 SmoothQuant：将激活异常值平滑到权重中
- 使用 AWQ：自动保护显著权重通道
- 使用 FP8：浮点格式天然处理大范围值

### 陷阱 2：校准数据不具代表性

**现象**：量化模型在基准测试上表现好，但在实际任务上表现差

**原因**：校准数据与实际使用分布不匹配

**解决方案**：
- 使用与实际任务分布一致的校准数据
- 校准集应包含多种主题、长度和复杂度
- 至少使用 128 个样本，复杂任务使用 256-512 个

### 陷阱 3：为小模型选择了过低的比特位宽

**现象**：7B 模型 2-bit 量化后几乎不可用

**原因**：小模型的冗余度低，无法承受激进量化

**解决方案**：
```
7B 模型：不低于 4-bit
13B 模型：不低于 3-bit
70B+ 模型：可以尝试 2-3 bit（配合 QAT）
```

### 陷阱 4：忽略 LayerNorm 和残差路径的精度

**现象**：量化后误差逐层累积，深层输出严重失真

**原因**：LayerNorm 和残差连接的精度对误差传播至关重要

**解决方案**：
- 保持 LayerNorm 权重在 FP16/FP32
- 残差路径使用全精度加法
- vLLM 的 `fused_add_rms_norm` 已正确处理此问题

### 陷阱 5：量化方法与硬件不匹配

**现象**：量化后推理速度反而变慢

**原因**：使用了硬件不支持的量化格式，导致软件模拟

**解决方案**：
- 严格按照第 7 节的硬件适配矩阵选择
- 使用 Marlin 内核（AWQ/GPTQ 可获得 3-10× 加速）
- 在目标硬件上实际测试性能

### 陷阱 6：只评估了单一基准

**现象**：MMLU 分数良好，但实际对话能力下降

**原因**：不同任务对量化的敏感度不同

**解决方案**：
- 至少使用 3 个以上基准评测（MMLU + GSM8K + HumanEval + 实际任务）
- 特别关注代码和推理任务
- 在实际业务数据上做 A/B 测试

### 陷阱 7：忽略 KV Cache 量化

**现象**：模型权重量化了但长上下文推理时仍然 OOM

**原因**：KV Cache 在长序列时占用显存可能超过模型本身

**解决方案**：
- 启用 FP8 KV Cache 量化
- 长上下文场景使用 KVQuant（3-bit）
- 考虑使用 PagedAttention 配合量化 KV Cache

### 陷阱 8：量化未充分训练的模型

**现象**：量化后精度不可逆地崩溃

**原因**：训练不足的模型权重分布不稳定，量化放大了未收敛的噪声

**解决方案**：
- 确保基础模型训练充分（> 100B tokens）
- 优先使用更大/训练更好的基础模型
- 避免量化早期 checkpoint

## 12. 基准测试与验证方法

### 12.1 标准评测基准

| 基准 | 类型 | 量化敏感度 | 说明 |
|------|------|-----------|------|
| **MMLU** | 知识问答 | 中 | 57 个学科的多选题 |
| **GSM8K** | 数学推理 | 低 | 小学数学题，量化下非常稳定 |
| **HumanEval** | 代码生成 | 低 | Python 代码生成，8-bit 下 99.9% 恢复 |
| **BBH** | 复杂推理 | 高 | 23 个困难任务 |
| **C-Eval** | 中文评测 | 中 | 中文知识能力评测 |
| **IFEval** | 指令遵循 | 中 | 指令遵循准确度 |
| **Arena-Hard** | 开放对话 | 中-高 | LLM 评审的对话质量 |

### 12.2 关键指标

**精度指标**：
- **Perplexity**：语言模型困惑度（基线指标）
- **Accuracy**：任务准确率
- **Pass@1**：代码生成首次通过率

**性能指标**：
- **Throughput**：输出 token 吞吐量（tok/s）
- **TTFT**：首 token 延迟（Time-To-First-Token）
- **显存占用**：峰值 GPU 显存使用

**语义相似度指标**：
- **ROUGE-1/L**：n-gram 重叠度
- **BERTScore**：上下文语义相似度
- **STS**：句子级语义相似度

### 12.3 验证流程

```
1. 建立基线
   └── 在 FP16 下运行所有基准，记录分数

2. 量化模型
   └── 使用选定的量化方法处理模型

3. 运行评测
   └── 在至少 3 个基准上评测量化模型

4. 对比分析
   ├── 计算每个基准的精度退化百分比
   ├── 如果退化 > 5%：调查原因
   └── 如果某个任务退化异常：可能需要调整量化策略

5. 性能测试
   ├── 测量吞吐量（不同 batch size）
   ├── 测量首 token 延迟
   └── 测量显存占用

6. 实际任务验证
   └── 在真实业务数据上进行 A/B 测试

7. 异常检查（红旗信号）
   ├── 8-bit 下 perplexity 增加 > 0.1 → 异常
   ├── 4-bit 下代码准确率下降 > 1% → 异常
   ├── GSM8K 下降 > 3% → 异常（该任务通常很稳定）
   └── 不同基准间方差过大 → 校准数据可能过拟合
```

## 13. 最佳实践检查清单

### 量化前

- [ ] 确认基础模型已充分训练（> 100B tokens）
- [ ] 确认目标硬件及其支持的量化格式
- [ ] 准备多样化的校准数据集（128-512 样本）
- [ ] 在 FP16 下建立精度基线
- [ ] 根据模型大小、硬件、精度要求选择量化方法
- [ ] 安装对应的量化工具（LLM Compressor / AutoAWQ / AutoGPTQ）

### 量化中

- [ ] 使用推荐的量化参数（group_size=128, bits=4 为常用配置）
- [ ] 保持 LayerNorm 和 lm_head 在全精度
- [ ] 对于 MoE 模型，使用专用 MoE 量化方法
- [ ] 如有 Hopper GPU，优先使用 FP8 W8A8
- [ ] 如使用 4-bit，优先选择 AWQ（精度优于 GPTQ）
- [ ] 启用 Marlin 内核加速（如适用）
- [ ] 长上下文场景启用 KV Cache 量化

### 量化后

- [ ] 在 3+ 个标准基准上评测（MMLU、GSM8K、HumanEval）
- [ ] 与 FP16 基线对比，精度退化在可接受范围内
- [ ] 在实际业务数据上验证
- [ ] 测量推理性能（吞吐量、延迟、显存占用）
- [ ] 检查红旗信号（异常退化、不同基准间方差过大）
- [ ] 如精度不达标：尝试更高比特或不同量化方法

### 部署

- [ ] 使用 vLLM 正确加载量化模型（指定 `--quantization` 参数）
- [ ] 配置适当的 `gpu-memory-utilization`
- [ ] 监控线上推理质量和性能
- [ ] 记录量化配置（方法、比特数、硬件、评测结果）
