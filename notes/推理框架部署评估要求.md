# 推理框架部署评估要求

## 一、评估目标

验证推理框架在生产环境的部署可行性，并在同等GPU配置下与NVIDIA A100进行性能对比评估。

---

### 2.1 测试模型类别

| 类别 | 说明 |
|------|------|
| 基座模型 | 大语言模型（LLM）、多模态模型 |
| 模型架构 | MoE（混合专家）、Dense、Hybrid等 |

### 2.2 测试组别

| 组别 | 描述 |
|------|------|

## 三、评估纬度与指标体系

### 3.1 推理性能

| 指标 | 条件 | 要求 |
|------|------|------|
| 吞吐量（Tokens/s） | 并发=16 | 达到 A100 同卡数的 **≥ 60%** |
| 请求吞吐（Req/s） | 并发=16 | 达到 A100 同卡数的 **≥ 60%** |

### 3.2 延迟表现

| 指标 | 条件 | 要求 |
|------|------|------|
| TTFT（Time To First Token） | 并发=16 | < 500ms |
| P99 TPOT（Time Per Output Token） | 并发=16 | < 100ms |
| ITL P99（Token间延迟） | 并发=16 | < 150ms |
| 端到端延迟 | 并发=16 | 视输出长度而定 |

### 3.3 并发扩展性

| 指标 | 要求 |
|------|------|
| 并发1→10→50→100 | 性能线性扩展或接近线性 |
| 突发流量恢复 | 100并发峰值后30s内恢复至正常响应 |

### 3.4 资源利用

| 指标 | 要求 |
|------|------|
| GPU显存占用 | 稳定状态下无泄漏 |
| GPU利用率 | ≥ 80%（推理负载下） |
| 多卡负载均衡 | TP/PP各卡负载均衡度≥90% |

### 3.5 集群稳定性

| 指标 | 要求 |
|------|------|
| 运行时间 | 7×24小时无故障 |
| 故障恢复 | 节点故障后5分钟内恢复服务 |
| 显存 | 无泄漏 |

**检查清单**：

| 检查项 | 操作 | 通过标准 |
|--------|------|----------|

### 4.3 功能测试

按第五章的测试点矩阵依次执行：基础推理（5.1）→ 高级功能（5.2）→ 长上下文（5.3）→ 多模态（5.4，如适用）→ 稳定性与边界（5.5）→ API 兼容性（5.6）。

**执行要求**：

### 4.4 性能测试

按第七章的性能测试矩阵执行，采集所有指标后与 A100/H100 基准数据对比。

**执行要求**：
- 使用 vLLM 内置 benchmark 工具（见 7.1 节）
- 每个测试点至少运行 **3 次**，取中位数
- 正式测试前发送 100+ 请求 **预热**，排除冷启动影响
- 测试期间同步采集 GPU/DCU/XPU 资源指标（见 4.4.1）
- 采样参数需在各平台间保持一致

#### 4.4.1 资源指标采集

测试期间需持续采集 GPU/DCU/XPU 的利用率与显存占用。各平台使用对应的监控命令：

```bash
# NVIDIA GPU — 每秒采样一次，输出到文件
nvidia-smi dmon -s pucvmet -d 1 -f gpu_metrics.csv

# Hygon DCU
hy-smi --loop 1 | tee dcu_metrics.log        # 或使用 rocm-smi（仅DCU）

# Kunlun XPU
xpu-smi dump -d 1 -t 0,1,2,3,4,5,6,7 | tee xpu_metrics.log
```

### 4.5 分布式测试

按第六章的详细方案执行多节点集群测试，包含：
- 多节点 PP（Pipeline Parallel）推理验证
- PD 分离架构测试
- LMCache 加速验证
- PD 分离 + LMCache 联合测试

**执行要求**：
- 先完成单节点全部测试并通过后，再进入分布式测试
- 记录跨节点通信带宽和延迟作为基线
- 多节点场景下需在每个节点上同步采集资源指标

### 4.6 稳定性测试

**执行流程**：

1. 以中等并发（16）启动持续压测，运行 **7×24 小时**
2. 每分钟采样一次显存占用，导出到 CSV 文件
3. 每小时记录一次吞吐量快照
4. 测试结束后对比首小时与末小时吞吐，衰减须 < 5%
5. 显存占用时序数据须绘图，确认无持续增长趋势

**显存监控脚本**：

```bash
#!/bin/bash
LOG_FILE="gpu_mem_stability_$(date +%Y%m%d_%H%M%S).csv"
echo "timestamp,gpu_index,mem_used_MiB,mem_total_MiB,gpu_util_pct" > "$LOG_FILE"
while true; do
    TS=$(date '+%Y-%m-%d %H:%M:%S')
    nvidia-smi --query-gpu=index,memory.used,memory.total,utilization.gpu \
        --format=csv,noheader,nounits | while read line; do
        echo "$TS,$line" >> "$LOG_FILE"
    done
    sleep 60
done
```

> DCU/XPU 平台须替换为对应的 `hy-smi` / `xpu-smi` 命令，输出格式保持一致。

**故障恢复测试**：

| 故障场景 | 操作方法 | 通过标准 |
|----------|----------|----------|
| 单进程 kill | `kill -9 <vllm_pid>` 后重启 | 5 分钟内恢复服务 |
| OOM 触发 | 发送超长请求触发显存不足 | 拒绝请求但服务不崩溃 |
| 网络闪断（多节点） | 短暂断开节点间网络 | 重连后自动恢复或明确报错 |

## 六、分布式场景测试要求

### 6.1 多节点基础验证

在进入高级分布式场景前，先验证多节点基础通信能力。

**前置条件**：
- 各节点间 RDMA/IB 网络已通
- 各节点已部署相同版本的 vLLM 和通信库
- 共享存储（NFS/Lustre）上存放模型权重，各节点可访问

**验证项**：

| 验证项 | 操作 | 通过标准 |
|--------|------|----------|
| 节点间网络带宽 | `ib_write_bw` 或 `perftest` | 带宽 ≥ 预期值（如 100Gb/s） |
| PP 跨节点推理 | 2 节点 PP=2 启动服务 | 模型加载成功、推理正常 |
| 多节点扩展 | 1→2→4 节点，相同负载测试吞吐 | 吞吐线性增长或接近线性 |
| 通信稳定性 | 持续推理 1 小时 | 无 hang、无 rank 失联 |

**多节点启动命令示例**（使用 Ray 或 torchrun）：

```bash
# 方式一：vLLM 原生多节点 (head 节点)
ray start --head --port=6379
python -m vllm.entrypoints.openai.api_server \
    --model <MODEL_PATH> \
    --tensor-parallel-size 8 \
    --pipeline-parallel-size 2 \
    --port 8000

# worker 节点
ray start --address=<HEAD_IP>:6379
```

### 6.2 PD分离（Prefill-Decode Separation）

PD 分离将 Prefill（预填充）和 Decode（解码）阶段分配到不同的节点组，以提升吞吐和资源利用率。

**测试架构**：

```
Client → Router/LB → Prefill 节点组 (≥1台) → KV Transfer → Decode 节点组 (≥1台) → Client
```

**测试内容**：

| 测试项 | 验证内容 | 通过标准 |
|--------|----------|----------|
| PD 架构部署 | Prefill 与 Decode 节点分别启动并通信 | 服务可用，推理结果正确 |
| 资源分配策略 | 不同 P:D 比例（1:1, 1:2, 2:1） | 记录各比例下的吞吐和延迟 |
| 负载均衡 | 多请求下 Prefill/Decode 节点负载分布 | 各节点 GPU 利用率差异 < 15% |
| PD 对比基线 | 同等卡数下 PD 分离 vs PD 合一 | 记录吞吐差异和 TTFT 差异 |
| KV Transfer 开销 | 观测 Prefill→Decode 的 KV Cache 传输耗时 | 传输延迟占总延迟比例 < 20% |

## 七、性能基准测试方法

### 7.1 测试工具

使用 vLLM 内置 benchmark 工具进行性能测试：

| 工具 | 用途 | 适用场景 |
|------|------|----------|
| `vllm bench serve` | 在线服务性能压测 | 模拟并发请求，测 TTFT/TPOT/吞吐 |
| `benchmark_serving.py` | 在线服务性能基准测试（旧版） | 同上，支持更多自定义参数 |
| `benchmark_throughput.py` | 离线吞吐量基准测试 | 不经过 HTTP，直接测最大吞吐 |
| `benchmark_latency.py` | 单请求延迟基准测试 | 测量无并发下的基线延迟 |

**标准压测命令模板**：

```bash
# 在线服务压测（推荐使用 vllm bench serve）
vllm bench serve \
    --base-url http://localhost:8000 \
    --model <MODEL_NAME> \
    --dataset-type random \
    --num-prompts <NUM> \
    --random-input-len <INPUT_LEN> \
    --random-output-len <OUTPUT_LEN> \
    --concurrency <N>

# 在线服务压测（使用 benchmark_serving.py）
python benchmarks/benchmark_serving.py \
    --backend vllm \
    --base-url http://localhost:8000 \
    --model <MODEL_NAME> \
    --dataset-name sharegpt \
    --dataset-path <SHAREGPT_JSON> \
    --num-prompts <NUM> \
    --request-rate <QPS>

# 离线吞吐测试
python benchmarks/benchmark_throughput.py \
    --model <MODEL_PATH> \
    --input-len <INPUT_LEN> \
    --output-len <OUTPUT_LEN> \
    --num-prompts <NUM> \
    --tensor-parallel-size <TP>

# 单请求延迟测试
python benchmarks/benchmark_latency.py \
    --model <MODEL_PATH> \
    --input-len <INPUT_LEN> \
    --output-len <OUTPUT_LEN> \
    --batch-size 1 \
    --tensor-parallel-size <TP>
```
